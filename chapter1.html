Lesson 1–1

LINE 平台架構全覽

在寫任何一行程式碼之前，必須先建立清晰的心智模型（mental model）： 當使用者在 LINE 傳一則訊息給你的 Bot，這個訊息是如何從手機抵達你的伺服器， 又是如何原路返回成為回覆？整個流程涉及三個主要角色。

LINE 使用者
手機 App
→
LINE Platform
雲端伺服器
→
你的 Webhook
POST /webhook
→
Reply API
HTTPS 回呼
→
LINE 使用者
收到回覆

LINE Platform 扮演的是中介者角色：它接收使用者的訊息，將其包裝成 JSON 格式的事件（event）， 以 HTTP POST 請求推送到你事先登記的 Webhook URL。 你的伺服器收到事件後處理邏輯，再呼叫 LINE 的 Reply API 傳回回覆。 整個過程是單向推送，不是你去輪詢（polling）LINE 有沒有新訊息。

這個 Webhook 架構有幾個重要含義。第一，你的伺服器必須對外可訪問， 本地開發時 localhost 並不符合這個條件，這就是為什麼稍後需要 Ngrok。 第二，你必須盡快回應 Webhook 請求（在 5 秒內回傳 HTTP 200）， 否則 LINE 會判定失敗並重試。若你的業務邏輯耗時較長（例如呼叫 AI API）， 應先回 200，再異步處理回覆。第三，LINE 會對每個 Webhook 請求附上數位簽名， 你必須驗證它的合法性。

架構補充 LINE 提供兩種主要的 Bot 產品類型：Messaging API（本課程重點） 與 LINE Login。Messaging API 用於建立自動化互動機器人； LINE Login 用於讓第三方網站以 LINE 身份登入，兩者可以搭配使用， 但服務目的不同，不應混淆。

了解這三個角色之後，我們再細看 LINE Platform 向你的 Webhook 發送的事件結構。 每個事件都包含以下核心欄位：

destination
接收事件的 Bot Channel ID，用於確認事件是發給你的 Bot。
events[ ]
事件陣列。LINE 可能在單一請求中批次傳送多個事件，你需要迭代處理每一個。
event.type
事件型別，如 message、follow、postback 等，決定後續處理邏輯。
event.source
事件來源，包含 type（user / group / room）與對應的 ID，讓你知道是誰觸發了事件。
event.replyToken
回覆用的一次性 token，每個 token 只能使用一次且於 30 秒內有效。僅部分事件型別含有此欄位。
event.timestamp
事件發生的 Unix 時間戳（毫秒），用於記錄或時序判斷。

Lesson 1–2

LINE Developers Console 設定

LINE Bot 的身份由兩個層次組成：Provider（供應商，代表你或你的組織） 與 Channel（具體的服務，每個 Bot 對應一個 Messaging API Channel）。 一個 Provider 可以建立多個 Channel。

前往 developers.line.biz，以你的 LINE 帳號登入。 若是公司用途，建議使用 LINE Business 帳號而非個人帳號， 避免日後帳號管理問題。
點選 Create a new provider，填入 Provider 名稱。 這個名稱會顯示在使用者的好友資訊頁，請填入正式的組織或品牌名稱。
在 Provider 下建立新 Channel，類型選 Messaging API。 填寫 Channel name（Bot 顯示名稱）、Channel description 與分類， 上傳 Bot 頭像圖片（建議 640×640px JPEG 或 PNG）。
Channel 建立後，進入 Basic settings 頁籤， 找到並複製 Channel secret（32位元十六進位字串）。 再進入 Messaging API 頁籤， 點選 Issue 產生 Channel access token (long-lived)。 兩組憑證都要妥善保存。
在 Messaging API 頁籤找到 Webhook settings， 啟用 Use webhook 選項， Webhook URL 欄位暫時留空，待後面步驟取得 Ngrok URL 後再填入。
建議在同一頁籤關閉 Auto-reply messages 和 Greeting messages，避免 LINE 自動回覆干擾你的 Webhook 測試。

完成設定後，你應該持有兩組關鍵字串：Channel secret 和 Channel access token。前者用於驗證 Webhook 請求合法性（接收端）， 後者用於呼叫 LINE API 傳送訊息（傳送端）。兩者性質不同，請勿混淆。

安全警告 這兩組憑證等同於你 Bot 的帳號密碼。絕對不能出現在 Git 提交記錄中， 不能寫死在程式碼裡，不能出現在截圖中發佈到公開討論區。 正確做法是存入 .env 檔案，並在 .gitignore 中排除此檔案。 一旦憑證外洩，應立即在 Console 重新產生（Reissue）。
Lesson 1–3

Node.js 專案初始化與 SDK 安裝

LINE 官方提供的 @line/bot-sdk 套件封裝了大量重複工作： Signature 驗證、型別定義、各種 Message Builder，以及 API 呼叫的錯誤處理。 雖然你也可以直接呼叫原始的 HTTP API，但使用 SDK 能減少出錯機會， 也讓程式碼更易讀。

建立專案目錄並安裝依賴
1# 建立並進入專案目錄
2mkdir my-linebot && cd my-linebot
3
4# 初始化 npm 專案（-y 跳過互動式提問）
5npm init -y
6
7# 安裝生產依賴
8npm install @line/bot-sdk express dotenv
9
10# 安裝開發工具（自動重啟）
11npm install --save-dev nodemon

接著建立 .env 檔案，存放所有敏感設定值。 這個檔案不應提交到版本控制，請同步建立 .gitignore。

.env — 環境變數設定
1# 從 LINE Developers Console 取得
2CHANNEL_SECRET=你的_channel_secret_32位
3CHANNEL_ACCESS_TOKEN=你的_long_lived_token
4
5# 伺服器設定
6PORT=3000
7NODE_ENV=development
.gitignore — 排除敏感檔案
1node_modules/
2.env
3.env.local
4.DS_Store

在 package.json 的 scripts 欄位加入開發用指令， 讓你每次修改程式碼後不需手動重啟伺服器：

package.json — scripts 欄位

4 "lint": "eslint ."
5}

最終你的專案根目錄應有如下結構。保持根目錄乾淨， 業務邏輯隨後會拆分到 handlers/ 與 services/ 等子目錄：

Lesson 1–4

Echo Bot 完整實作

Echo Bot 是最小可運行的 LINE Bot：用戶說什麼，Bot 就回什麼。 這個看似簡單的實作包含了 Bot 開發的所有核心概念： Express 伺服器、Middleware 驗證、事件解析、Reply API 呼叫。 先把這個框架搞清楚，後面的章節只是在它之上堆疊功能。

index.js — 完整 Echo Bot 實作
1require('dotenv').config()
2const line = require('@line/bot-sdk')
3const express = require('express')
4
5// 1. 設定物件：SDK 與 Client 都需要它
6const config = {
7 channelSecret: process.env.CHANNEL_SECRET,
8 channelAccessToken: process.env.CHANNEL_ACCESS_TOKEN,
9}
10
11// 2. LINE Client：用於呼叫 LINE API（傳送訊息等）
12const client = new line.Client(config)
13
14// 3. Express App
15const app = express()
16
17// 4. Webhook 端點：line.middleware 負責驗證簽名並解析 body
18app.post('/webhook', line.middleware(config), (req, res) => {
19 Promise
20 .all(req.body.events.map(handleEvent))
21 .then(result => res.json(result))
22 .catch(err => {
23 console.error(err)
24 res.status(500).end()
25 })
26})
27
28// 5. 事件處理函式：處理單一 event 物件
29async function handleEvent(event) {
30 // 只處理文字訊息，其他事件型別一律回傳 null
31 if (event.type !== 'message' || event.message.type !== 'text') {
32 return null
33 }
34
35 // 取得使用者傳送的文字
36 const userText = event.message.text
37
38 // 呼叫 Reply API，以 replyToken 回覆
39 return client.replyMessage(event.replyToken, {
40 type: 'text',
41 text: `你說：${userText}`,
42 })
43}
44
45// 6. 健康檢查端點（部署監控用）
46app.get('/health', (_req, res) => res.json({ status: 'ok' }))
47
48// 7. 啟動伺服器
49const PORT = process.env.PORT || 3000
50app.listen(PORT, () => console.log(`Bot running on port ${PORT}`))

程式碼第 18 行的 line.middleware(config) 是這個架構的關鍵。 它是一個 Express middleware，在你的處理邏輯執行之前先做兩件事： 一是從請求的 x-line-signature 標頭取出簽名， 以 HMAC-SHA256 演算法搭配你的 Channel Secret 重新計算， 比對兩者是否一致，不一致直接回傳 403； 二是解析 JSON body，填入 req.body。 這兩件事由 SDK 處理，你不需要手動實作。

第 20 行使用 Promise.all() 是因為 LINE 可能在一次請求中批次傳送多個事件， 必須對 events 陣列的每一個元素分別呼叫 handleEvent， 並等待全部完成。第 32 行對非文字訊息回傳 null， 這是正確做法——告訴 Promise.all 這個事件不需要處理， 不應對其呼叫 replyMessage（因為 sticker、image 等事件也可能沒有 replyToken）。

常見錯誤 如果你在 handleEvent 中忘記對不需要回覆的事件 return null， 而是讓函式自然結束（回傳 undefined），在某些情況下 Promise.all 會拋出例外，導致 500 錯誤。養成明確 return null 的習慣。
Lesson 1–5

Ngrok 本地測試與 Webhook 驗證

你的伺服器跑在 localhost:3000，LINE 的雲端伺服器無法直接存取它。 Ngrok 的作用是在你的本地端建立一條安全隧道（tunnel）， 並提供一個公開的 HTTPS URL 指向你的本地服務。 這個 URL 的流量會透過 Ngrok 的伺服器轉發到你的電腦。

安裝並啟動 Ngrok
1# 方式一：全域安裝（需先在 ngrok.com 建立免費帳號）
2npm install -g ngrok
3ngrok config add-authtoken 你的_authtoken
4
5# 方式二：直接從 ngrok.com 下載執行檔（免安裝）
6
7# 啟動隧道，對外暴露 3000 port
8ngrok http 3000

啟動後，終端機會顯示類似下方的輸出。記下 Forwarding 那一行的 HTTPS URL：

Ngrok 啟動輸出
1Session Status online
2Account [email protected] (Plan: Free)
3Version 3.x.x
4Region Asia Pacific (ap)
5Forwarding https://a1b2c3d4e5f6.ngrok-free.app -> http://localhost:3000
6
7Connections ttl=0 opn=0 rt1=0.00 rt5=0.00 p50=0.00 p90=0.00

將這個 HTTPS URL 加上 /webhook，填入 LINE Developers Console 的 Webhook URL 欄位。 例如：https://a1b2c3d4e5f6.ngrok-free.app/webhook。 填入後點選 Verify 按鈕，LINE 會發送一個測試請求到你的 URL， 若返回 200 則表示設定正確。

重要限制 Ngrok 免費方案的 URL 在每次重啟後都會更換。 每次重啟 Ngrok 後，必須回到 LINE Console 更新 Webhook URL。 這在開發階段是正常工作流程，不影響最終部署（正式部署時使用固定域名）。 Ngrok 也提供付費方案可綁定固定子域名，有需要可以考慮。

完成後，在另一個終端機視窗啟動你的 Bot 伺服器：

啟動本地開發伺服器
1# 開發模式（檔案變更自動重啟）
2npm run dev
3
4# 終端機應顯示：
5Bot running on port 3000

現在用你的個人 LINE 帳號掃描 Developers Console 中的 Bot QR Code， 加好友後傳一則訊息，應該能看到 Bot 回覆「你說：XXXXX」。 Ngrok 的終端機介面也會即時顯示每個 HTTP 請求的狀態碼， 方便調試。

章節測驗
當 LINE Bot 需要主動在 30 分鐘後推送通知給用戶（非回覆），應使用哪種方式？
A. 保存 replyToken 延後使用
B. 呼叫 Push API，傳入 userId
C. 重新觸發一個 Webhook 事件
D. 使用 Reply API 搭配 setTimeout
正確。replyToken 有效期僅 30 秒，且為一次性，不可延後使用。 主動推送必須使用 Push API，傳入預先儲存的 userId， 這也是為什麼在 follow 事件時要將 userId 存入資料庫。
