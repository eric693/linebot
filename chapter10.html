Lesson 10–2

用戶旅程與事件路由的完整實作

電商客服 Bot 的用戶從加好友那一刻開始， 就進入一個需要系統性管理的旅程： 引導綁定、使用選單功能、發起 AI 諮詢、查詢訂單、轉人工客服。 每個節點的行為取決於用戶目前的狀態（是否已綁定、目前的 FSM 狀態）。

完整用戶旅程
Step 1
加好友 — Follow 事件
記錄用戶到資料庫（findOrCreate），發送歡迎訊息（含綁定引導按鈕）， 設定訪客版 Rich Menu（RICH_MENU_GUEST）。
Step 2
點選「綁定帳號」— Postback 事件
傳送含 LIFF URL 的按鈕訊息，用戶點擊後開啟帳號綁定頁面。 LIFF 完成後觸發後端 /api/liff/bind，切換為會員版 Rich Menu。
Step 3
使用 Rich Menu — 選單互動
會員版選單提供「查訂單」、「AI 客服」、「專人服務」三個熱區， 各自觸發不同的 Postback action。
Step 4A
查訂單 — Postback + Flex Message
查詢該用戶最近 3 筆訂單，以 Flex Message Carousel 展示， 每張卡片包含訂單狀態、物流號碼和「查看詳情」按鈕。
Step 4B
AI 客服 — 多輪對話模式
進入 AI 對話模式，所有文字訊息路由到 AI handler， 保有多輪記憶，可查詢訂單（Function Calling）。 傳送「/離開」結束模式並清除記憶。
Step 4C
轉人工客服 — FSM 流程
進入 SUPPORT FSM，收集問題描述， 推播通知到客服管理群組，用戶收到「已轉接，客服將於 X 分鐘內回覆」確認訊息。

人工客服 FSM

當用戶選擇「專人服務」，Bot 進入客服收集流程： 詢問問題類型（Buttons Template）、收集問題描述， 最後推播 Push Message 到客服群組通知有新案件， 同時告知用戶預計回覆時間。

src/fsm/supportFSM.js — 人工客服收集流程
1const SUPPORT_FSM = {
2AWAIT_ISSUE_TYPE: {
3'ORDER_ISSUE': { next: 'AWAIT_DESCRIPTION', ctx: 'order' },
4'PRODUCT_ISSUE': { next: 'AWAIT_DESCRIPTION', ctx: 'product' },
5'OTHER': { next: 'AWAIT_DESCRIPTION', ctx: 'other' },
6'CANCEL': { next: 'DONE', action: 'cancel' },
7},
8AWAIT_DESCRIPTION: {
9'TEXT_INPUT': { next: 'DONE', action: 'submit' },
10},
11}
12
13async function handleFSMMessage(event, client) {
14const userId = event.source.userId
15const userText = event.message.text
16const session = await SessionStore.get(userId, 'support-fsm')
17const state = session.state
18const ctx = session.context || {}
19
20// 取得當前狀態的轉換表
21const transitions = SUPPORT_FSM[state]
22const eventKey = transitions[userText] ? userText : 'TEXT_INPUT'
23const transition = transitions[eventKey]
24if (!transition) return
25
26// 更新 context
27if (transition.ctx) ctx.issueType = transition.ctx
28if (eventKey === 'TEXT_INPUT') ctx.description = userText
29
30if (transition.action === 'submit') {
31await submitSupportRequest(userId, ctx, client)
32await SessionStore.clear(userId, 'support-fsm')
33} else {
34await SessionStore.set(userId, 'support-fsm', transition.next, ctx)
35await sendNextPrompt(userId, transition.next, client)
36}
37}

Lesson 10–4

整合測試策略與關鍵業務流程驗證

LINE Bot 的整合測試面臨一個核心問題： 真實測試需要呼叫 LINE API、OpenAI API 和 Redis， 這既昂貴又不穩定，也無法在 CI 中自動化運行。 解法是測試替身（Test Double）—— 用假的實作替換外部依賴， 讓你能在不連網路的環境中驗證業務邏輯的正確性。

測試策略分層
層級測試範圍速度工具
單元測試 Flex Message 範本、FSM 轉換邏輯、工具函式 極快（ms） Jest
整合測試 Webhook 端點 + 資料庫（使用 SQLite in-memory） 快（秒） Jest + supertest
端對端測試 真實 LINE API + 測試 Channel 慢（分鐘） 手動 / 專屬測試腳本

Lesson 10–5

上線前確認清單與課程總結

第一次將 Bot 切換到真實用戶流量之前， 有一系列容易被忽略的設定需要逐一確認。 跳過這些步驟不會讓 Bot 無法運作， 但可能在高流量時造成意外費用、在問題發生時缺乏診斷資訊、 或在資安事件中讓損失擴大。

完成這十章後，你已具備從零開始獨立開發、部署、 並維運一個生產級 LINE Bot 的完整能力。 以下是這套系統的核心設計決策， 每一個都可以在你的下一個專案中根據需求調整：

事件驅動架構
Webhook 立即回應 200，事件非同步處理， 確保在任何部署環境下都不會因為業務邏輯複雜度超過 LINE 的 1 秒限制而失敗。
狀態感知路由
EventRouter 在分流前先查詢 Redis session， FSM 模式優先於 AI 模式優先於一般訊息， 讓多個進行中的流程（帳號綁定、客服收集）能並行運作而不互相干擾。
Repository 模式
所有資料庫操作封裝在 Repository 類別中， 業務邏輯不直接依賴 Knex 查詢語法， 讓測試替換資料層、未來更換資料庫引擎都不影響上層程式碼。
費用邊界明確
每一個會產生外部費用的呼叫（OpenAI、LINE Push Message） 都有對應的速率限制或去重機制， 且都有可觀測的指標計數，讓費用異常能被及早發現。
防禦縱深
Webhook 簽名驗證 + 全局 IP 速率限制 + 每用戶 AI 速率限制 + Prompt Injection 過濾， 四層防護各自獨立，任一層失效不影響其他層的保護作用。
可觀測性優先
結構化日誌（pino）、集中化錯誤追蹤（Sentry）、 外部監控（Betteruptime）和應用程式指標共同構成完整的可觀測性基礎， 確保問題發生時能快速定位根本原因。
延伸方向 本課程建立了 Bot 開發的核心基礎，以下是進階延伸的學習方向。 若需要服務大量同時在線用戶，可研究 Node.js Cluster 模式或 使用 PM2 的多核心模式水平擴展。 若需要更豐富的 AI 知識庫支援，可以整合 pgvector 和 RAG 架構， 讓 Bot 能夠回答基於你的內部文件的問題。 若需要複雜的推廣活動管理（集點、抽獎、分眾推播）， 可以在現有資料架構上擴充 campaigns 和 segments 資料表。 所有這些延伸都建立在本課程建立的架構基礎之上， 不需要推翻現有設計。

EventRouter 為何讓 FSM 模式的優先級高於 AI 模式？
A.FSM 的程式碼比 AI handler 更輕量，可以先執行以節省資源
B.FSM 是有限期的表單收集流程，若被 AI 搶先處理會導致流程中斷
C.AI handler 需要呼叫外部 API，延遲高，應排在後面
D.LINE 平台規定 Postback 事件必須優先處理
正確。FSM 代表一個進行中的多步驟流程（如客服問題收集）， 這個流程有明確的開始和結束，每個步驟都在等待特定的用戶輸入。 若用戶在 FSM 進行中傳送的任意文字被路由到 AI 而不是 FSM， FSM 永遠無法收到它等待的輸入，流程將永久卡住。 相反地，AI 對話模式沒有嚴格的步驟依賴， 被 FSM 中斷後用戶完成 FSM 流程再繼續 AI 對話也完全合理。
