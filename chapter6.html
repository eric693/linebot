資料庫整合
與狀態管理

無記憶的 Bot 只能處理單輪對話，無法追蹤用戶的累積行為、偏好或對話進度。 本章建立 Bot 的持久化記憶層：從選擇資料庫、設計 schema、實作 Repository 模式， 到建構多輪對話的有限狀態機。完成後你的 Bot 將能記住每位用戶是誰、 他們做過什麼，以及他們目前在哪個對話流程中。

學習目標
  • 評估 SQLite 與 PostgreSQL 的適用場景
  • 用 Knex.js 建立 migration 管理資料庫版本
  • 設計用戶與對話記錄的資料表結構
  • 實作 Repository 模式封裝資料存取邏輯
  • 建立有限狀態機管理多輪對話流程
  • 用 Redis 實作高效能 session 狀態儲存
Lesson 6–1

資料庫選型：SQLite vs PostgreSQL vs Redis

大多數 LINE Bot 需要三種不同性質的儲存需求， 而沒有任何一種資料庫能同時以最佳方式滿足全部需求。 理解每種需求的特性，才能做出正確的技術選型， 避免後期因選錯而付出高昂的重構代價。

三種儲存需求
持久化業務資料
用戶帳號、訂單記錄、點數歷史——需要長期保存、支援複雜查詢的關係型資料。 適合 PostgreSQL（生產）或 SQLite（開發）。
對話狀態（Session）
用戶目前在哪個對話步驟、暫存中的輸入值——存取頻率極高， 需要毫秒級讀寫，對話結束後可以丟棄。 適合 Redis，搭配 TTL 自動清理過期 session。
快取
頻繁查詢但不常變動的資料（商品列表、設定檔）。 用 Redis 快取資料庫查詢結果，減少資料庫壓力。
SQLite：開發環境首選

SQLite 是嵌入式資料庫，整個資料庫就是磁碟上的一個 .db 檔案， 無需安裝伺服器，npm install 後即可使用。 對於本地開發、功能驗證和小型部署（日活躍用戶 1,000 以下）而言， SQLite 的簡單性遠超其他選項的優勢。唯一的核心限制是 不支援並發寫入——同一時間只能有一個寫入操作， 在大量并行 Webhook 請求下會成為瓶頸。

PostgreSQL：生產環境標準

PostgreSQL 支援完整的 ACID 事務、並行寫入、jsonb 欄位、 全文搜尋和豐富的索引類型。本章所有範例使用 Knex.js 作為 Query Builder， 讓相同程式碼同時支援 SQLite（開發）和 PostgreSQL（生產）， 只需修改 knexfile.js 中的連線設定。

特性 SQLite PostgreSQL Redis
安裝複雜度 無需安裝 需要伺服器 需要伺服器
並發寫入 不支援 支援 支援（單執行緒）
資料持久性 磁碟檔案 磁碟 預設記憶體（可持久化）
TTL 支援 需手動實作 需手動實作 原生 EXPIRE
典型讀取延遲 0.1–1 ms 1–10 ms 0.1–0.3 ms
適合用途 開發、小型部署 生產業務資料 Session、快取
推薦架構 開發階段：SQLite + Redis（本地 Docker）。生產環境：PostgreSQL + Redis。 使用 Knex.js 抽象資料庫差異，讓兩套環境共用業務邏輯， 只有 knexfile.js 因環境不同而有差異。
Lesson 6–2

Knex.js Migration 與資料表設計

Migration 是資料庫結構變更的版本控制機制：每一次對資料表的修改 都記錄為一個帶有時間戳記的檔案，可以被追蹤、執行（up）、 還原（down），讓多人協作和環境同步有明確的操作基準。 沒有 migration 機制直接修改生產資料庫是極度危險的做法。

Lesson 6–3

Repository 模式：封裝資料存取邏輯

Repository 模式的核心思想：業務邏輯不應該知道資料如何被儲存。 你的 Webhook handler 知道「要取得這個用戶的訂單」， 但不應該知道「這件事需要執行一個帶有 JOIN 和 WHERE 子句的 SQL 查詢」。 將所有資料庫操作封裝在 Repository 類別中，帶來三個直接好處： 測試時可以用 mock 替換整個 Repository、 更換底層資料庫只需修改 Repository 而不影響其他程式碼、 SQL 邏輯集中管理不分散在各處。

src/repositories/UserRepository.js
1const db = require('../../db/knex') // knex 單例
2
3class UserRepository {
4
5// 以 LINE userId 查詢，最常被呼叫的方法
6async findByLineId(lineUserId) {
7return db('users')
8.where({ line_user_id: lineUserId })
9.first() // 回傳第一筆或 undefined（不是陣列）
10}
11
12// 查詢不到就建立——Webhook 中極常用的模式
13async findOrCreate(lineUserId, defaults = {}) {
14let user = await this.findByLineId(lineUserId)
15if (!user) {
16[user] = await db('users')
17.insert({ line_user_id: lineUserId, ...defaults })
18.returning('*') // PostgreSQL；SQLite 需另查（見後文）
19}
20return user
21}
22
23// 更新部分欄位，自動設定 updated_at
24async update(lineUserId, patch) {
25return db('users')
26.where({ line_user_id: lineUserId })
27.update({ ...patch, updated_at: db.fn.now() })
28}
29
30// 原子性增加點數（避免讀取–修改–寫入的競態條件）
31async incrementPoints(lineUserId, amount) {
32return db('users')
33.where({ line_user_id: lineUserId })
34.increment('points', amount)
35// SQL: UPDATE users SET points = points + ? WHERE line_user_id = ?
36}
37}
38
39module.exports = new UserRepository() // 匯出單例

SQLite 相容性注意 Knex 的 .returning('*') 在 PostgreSQL 上直接回傳插入的資料列， 但舊版 SQLite 不支援 RETURNING 子句。 需同時支援兩種資料庫時，SQLite 環境下改為先 insert 取得 lastID， 再以 id 查詢一次，即可相容。
Lesson 6–4

有限狀態機：管理多輪對話流程

當 Bot 需要引導用戶完成多步驟流程——「收集姓名 → 收集電話 → 確認 → 完成」—— 最常見的錯誤做法是在 message handler 裡用大量 if/else 判斷 「用戶上一句說了什麼」。這種方法在流程超過三個步驟後就難以維護。

有限狀態機（FSM） 是解決這個問題的經典模式。 系統在任何時刻只處於一個狀態， 收到輸入後，依據預先定義的轉移規則， 決定轉移到哪個新狀態並執行對應的動作。 對話流程中，每個「等待用戶輸入」的步驟就是一個狀態；每則訊息就是一個輸入。

FSM Engine：通用執行器

狀態機的定義與執行邏輯應該分離。 以下通用的 FSMEngine 接受任意狀態機定義、當前狀態、輸入事件， 執行轉移並回傳新狀態與更新後的 context。

src/fsm/FSMEngine.js
1class FSMEngine {
2
3/**
4 * 執行一次狀態轉移
5 * @returns {{ newState, context, handled }}
6 * handled = false 表示無對應轉移規則
7 */
8async dispatch(fsm, currentState, context, eventType, input, client, replyToken, userId) {
9const stateDef = fsm[currentState]
10if (!stateDef) return { newState: currentState, context, handled: false }
11
12const transition = stateDef.transitions[eventType]
13if (!transition) return { newState: currentState, context, handled: false }
14
15// 執行 guard（驗證輸入）
16if (transition.guard && !transition.guard(input)) {
17await transition.onGuardFail(context, client, replyToken)
18return { newState: currentState, context, handled: true } // 狀態不變
19}
20
21// 執行 action（傳送回覆、寫入資料庫等）
22if (transition.action) {
23await transition.action(context, client, replyToken, input, userId)
24}
25
26return {
27newState: transition.next,
28context, // context 是可變物件，action 已就地修改
29handled: true,
30}
31}
32
33// 將文字輸入正規化為 FSM 事件類型
34classifyTextInput(text) {
35const norm = text.trim().toLowerCase()
36if (['確認', 'confirm', 'yes', 'y'].includes(norm)) return 'CONFIRM'
37if (['重新', '重填', 'restart', 'no'].includes(norm)) return 'RESTART'
38if (norm === '取消') return 'CANCEL'
39return 'TEXT_INPUT'
40}
41}
42
43module.exports = new FSMEngine()
Lesson 6–5

Redis Session 儲存與 FSM 整合

FSM 的狀態（currentState 和 context） 必須在兩次 Webhook 請求之間持久保存，但不需要保存太久。 將 session 存入 Redis 有兩個核心優勢： 讀寫延遲約 0.1–0.3 ms，遠快於關聯式資料庫； 可為每個 session 設定 TTL（如 30 分鐘後自動過期）， 用戶放棄流程後無需手動清理。

src/session/RedisSessionStore.js
1const Redis = require('ioredis') // npm install ioredis
2
3const redis = new Redis(process.env.REDIS_URL || 'redis://localhost:6379')
4const SESSION_TTL = 60 * 30 // 30 分鐘（秒為單位）
5
6class RedisSessionStore {
7
8// Key 格式：session:{lineUserId}:{fsmName}
9// fsmName 讓同一用戶可並行跑多個不同流程
10_key(userId, fsmName) {
11return `session:${userId}:${fsmName}`
12}
13
14// 取得 session；若不存在回傳預設值
15async get(userId, fsmName, initialState = 'IDLE') {
16const raw = await redis.get(this._key(userId, fsmName))
17if (!raw) return { state: initialState, context: {} }
18return JSON.parse(raw)
19}
20
21// 儲存 session，重設 TTL
22async set(userId, fsmName, state, context) {
23await redis.setex(
24this._key(userId, fsmName),
25SESSION_TTL,
26JSON.stringify({ state, context }),
27)
28}
29
30// 流程結束後手動清除（不等 TTL）
31async clear(userId, fsmName) {
32await redis.del(this._key(userId, fsmName))
33}
34}
35
36module.exports = new RedisSessionStore()
在 Message Handler 中整合完整流程
src/handlers/messageHandler.js — 整合 FSM + Session + Repository
1const SessionStore = require('../session/RedisSessionStore')
2const FSMEngine = require('../fsm/FSMEngine')
3const { REGISTRATION_FSM } = require('../fsm/registrationFSM')
4const UserRepo = require('../repositories/UserRepository')
5const MessageRepo = require('../repositories/MessageRepository')
6
7async function handleMessage(event, client) {
8const userId = event.source.userId
9const text = event.message.type === 'text' ? event.message.text : ''
10
11// 1. 取得或建立用戶記錄
12const user = await UserRepo.findOrCreate(userId)
13
14// 2. 非同步記錄訊息（不阻塞主流程）
15MessageRepo.log(user.id, 'inbound', event.message.type, text, event)
16.catch(err => console.error('MessageLog failed:', err))
17
18// 3. 取得此用戶的 registration FSM session
19const session = await SessionStore.get(userId, 'registration')
20
21// 4. 用戶發送「開始註冊」，觸發 START_REGISTER
22if (text === '開始註冊' && session.state === 'IDLE') {
23const result = await FSMEngine.dispatch(
24REGISTRATION_FSM, 'IDLE', {}, 'START_REGISTER',
25null, client, event.replyToken, userId,
26)
27await SessionStore.set(userId, 'registration', result.newState, result.context)
28return
29}
30
31// 5. 正在 FSM 流程中，繼續處理
32if (session.state !== 'IDLE' && session.state !== 'DONE') {
33const eventType = FSMEngine.classifyTextInput(text)
34const result = await FSMEngine.dispatch(
35REGISTRATION_FSM, session.state, session.context,
36eventType, text, client, event.replyToken, userId,
37)
38if (result.newState === 'DONE') {
39await SessionStore.clear(userId, 'registration') // 完成，清除 session
40} else {
41await SessionStore.set(userId, 'registration', result.newState, result.context)
42}
43return // FSM 處理完畢，不執行一般訊息邏輯
44}
45
46// 6. 非 FSM 流程中的一般訊息回覆
47return client.replyMessage(event.replyToken, {
48type: 'text',
49text: `您好 ${user.display_name || ''}，請使用下方選單或輸入「開始註冊」完成會員設定。`,
50})
51}

無 Redis 的備選方案 若環境不便安裝 Redis，可將 session 存入資料庫的 conversation_states 表。 讀寫延遲會增加，但功能邏輯完全相同——這是 Repository 模式的價值： 把 RedisSessionStore 換成 DBSessionStore， handleMessage 中的程式碼完全不需要修改。

核心術語
Migration
資料庫結構變更的版本化記錄，可執行（up）與還原（down）。
Repository Pattern
將所有資料存取封裝在類別中，業務邏輯不直接操作資料庫。
FSM
有限狀態機：系統在有限個狀態間依轉移規則切換，管理多輪對話的標準模式。
Guard
FSM 轉移前的驗證函式，輸入不合法則停留在當前狀態。
TTL
Time To Live，Redis 鍵值的過期時間，session 到期後自動刪除。
Connection Pool
預先建立並維護多條資料庫連線，避免每次請求重新握手的開銷。
Atomic Operation
不可被打斷的單一操作。SQL 的 INCREMENT 原子執行，避免並發競態。

當 FSM 的 guard 函式回傳 false 時，正確的行為是？
A.轉移到 next 狀態，同時執行 onGuardFail
B.執行 onGuardFail，狀態保持不變
C.轉移到 IDLE，重設 context
D.拋出例外，由呼叫端處理
正確。Guard 失敗代表輸入不合法（如電話格式錯誤）， 應提示用戶重新輸入，停留在當前狀態繼續等待下一次有效輸入， 而不是轉移到其他狀態或拋出錯誤。
