訊息型別與
Reply / Push API

Bot 能傳送的遠不只是文字。本章系統性地覆蓋所有可傳送的訊息型別—— 從圖片、影片、位置，到按鈕樣板、輪播卡片——並深入探討 Reply API 與 Push API 在費用、時效與使用情境上的根本差異。 選錯 API 不只影響成本，更直接影響用戶體驗。

學習目標
  • 掌握全部可傳送訊息型別的結構與限制
  • 理解 Reply、Push、Multicast、Broadcast 的差異與費用
  • 實作 Buttons Template 與 Confirm Template
  • 實作 Carousel Template 多欄輪播卡片
  • 設計 Quick Reply 快速回覆選項
  • 制定符合實際需求的訊息推送策略
Lesson 3–1

所有可傳送訊息型別總覽

LINE Bot 能傳送的訊息型別分為兩大類：基本訊息（Basic Messages） 與樣板訊息（Template Messages）。 基本訊息對應使用者日常能傳送的格式；樣板訊息是 Bot 專屬的結構化介面， 提供按鈕、輪播等互動元件，但只在行動版 LINE 上完整呈現， 桌機版或不支援的用戶端會顯示 altText 替代文字。 第四章會介紹功能更強大的 Flex Message，本章聚焦於基本訊息與樣板訊息。

類別 type 值 關鍵欄位 使用場景
基本 text text 一般對話、通知、確認訊息，支援換行與 LINE emoji
基本 image originalContentUrl, previewImageUrl 商品圖、活動橫幅、報告截圖，需提供公開可訪問的 HTTPS URL
基本 video originalContentUrl, previewImageUrl 教學影片、產品展示，需同時提供預覽縮圖
基本 audio originalContentUrl, duration 語音通知，duration 為毫秒整數
基本 location title, address, latitude, longitude 門市地址、集合地點、外送位置
基本 sticker packageId, stickerId 互動回應、情緒表達，僅限 LINE 官方貼圖包
樣板 template / buttons title, text, actions 單一選單、確認提示，最多 4 個按鈕動作
樣板 template / confirm text, actions（僅 2 個） 是／否確認、選擇操作，強制兩選一
樣板 template / carousel columns（最多 10 欄） 商品列表、方案比較、多項目選擇
樣板 template / image_carousel columns（僅圖片 + 單一 action） 純圖片輪播，視覺為主的情境

所有傳送媒體（image、video、audio）的 URL 都必須是 HTTPS， 且必須能從公開網路直接訪問——LINE 的伺服器會主動去抓取這些資源以產生預覽。 本地端的 localhost URL 不可用，開發期間必須先將媒體上傳到雲端或使用 CDN。 圖片的 previewImageUrl 通常是原圖的壓縮版本， 建議寬度不超過 240px 以加快預覽載入速度， 而 originalContentUrl 則是點擊後顯示的完整原圖。

一次最多傳送 5 則

Reply API 與 Push API 的 messages 參數都接受陣列， 一次呼叫最多可傳送 5 則訊息。 這讓你可以組合不同型別，例如先傳文字說明、再傳圖片、最後附上按鈕選單， 在一次 API 呼叫中完成整個互動單元，減少 API 呼叫次數與用戶等待感。

一次傳送多種型別的組合範例
1await client.replyMessage(event.replyToken, [ 2 // 訊息一：文字說明 3 { 4 type: 'text', 5 text: '本月最新優惠方案如下：' 6 }, 7 // 訊息二：圖片 8 { 9 type: 'image', 10 originalContentUrl: 'https://cdn.example.com/promo-may.jpg', 11 previewImageUrl: 'https://cdn.example.com/promo-may-thumb.jpg' 12 }, 13 // 訊息三：位置（門市地址） 14 { 15 type: 'location', 16 title: '台北旗艦店', 17 address: '台北市信義區松高路12號', 18 latitude: 25.0408, 19 longitude: 121.5676 20 } 21])
Lesson 3–2

Reply API 與 Push API：費用、時效與選擇策略

LINE Messaging API 提供四種不同的訊息傳送方式，各有其對象、費用結構與使用時機。 混淆這四者是初學者最常犯的架構錯誤，選擇不當會導致不必要的費用支出， 或在需要主動通知時才發現無法傳送。

API
對象
費用
觸發條件
Reply API
單一用戶，需 replyToken
免費，不計入訊息用量
必須由 Webhook 事件觸發，30 秒內有效
Push API
單一用戶，需 userId
計費，依方案訊息則數計算
可在任意時間主動觸發，無需用戶互動
Multicast API
最多 500 個 userId 清單
計費，依訊息則數 × 用戶數
批次推送，適合分群行銷、分批通知
Broadcast API
所有追蹤者（無需 userId 清單）
計費，依總追蹤人數計算
全員公告、促銷活動，注意費用可能極高

理解這個表格後，決策規則很直觀：只要有 replyToken 就優先用 Reply API， 因為它完全免費且天然契合互動語境。需要主動推送時才使用 Push API， 並務必在資料庫中事先儲存好 userId（通常在 follow 事件時儲存）。 Multicast 適合同時通知一批特定用戶（例如「已下訂單的所有用戶」）， Broadcast 則適合真正全員公告，使用前請確認你的 LINE Official Account 訂閱方案的訊息額度。

四種 API 的程式碼對照
四種傳送方式的呼叫語法
1// Reply API：需要 replyToken，只能用一次，30 秒有效 2await client.replyMessage(event.replyToken, message) 3 4// Push API：需要 userId（或 groupId / roomId） 5await client.pushMessage(userId, message) 6 7// Multicast：userId 陣列，上限 500，訊息為單一物件或陣列 8await client.multicast([userId1, userId2, userId3], message) 9 10// Broadcast：無需提供用戶清單，推送給所有追蹤者 11await client.broadcast(message) 12 13// Multicast 超過 500 人時需分批 14async function multicastAll(allUserIds, message) { 15 const BATCH = 500 16 for (let i = 0; i < allUserIds.length; i += BATCH) { 17 const batch = allUserIds.slice(i, i + BATCH) 18 await client.multicast(batch, message) 19 // 批次之間稍作等待，避免 rate limit 20 await new Promise(r => setTimeout(r, 200)) 21 } 22}
費用警告 LINE Official Account 的訊息費用依方案而異。免費方案每月有限額， 超出後每則訊息都會計費。Broadcast 對 10,000 名追蹤者傳送 1 則訊息， 就消耗 10,000 則配額。在傳送大量推播前，務必先確認當月剩餘額度， 或升級到適合的付費方案，避免月底超額停送。 可透過 client.getMessageQuota() 取得當月額度與已用量。
查詢訊息配額
取得訊息配額與已使用量
1async function checkQuota() { 2 // 取得本月配額上限 3 const quota = await client.getMessageQuota() 4 console.log('配額上限：', quota.value) // -1 代表無限制 5 6 // 取得本月已傳送則數 7 const usage = await client.getMessageQuotaConsumption() 8 console.log('已使用：', usage.totalUsage) 9 10 const remaining = quota.value - usage.totalUsage 11 console.log(`剩餘：${remaining} 則`) 12 return remaining 13}
Lesson 3–3

Buttons Template 與 Confirm Template

Template Messages 讓 Bot 傳送帶有可點擊按鈕的結構化訊息， 是製作互動選單的基礎。按鈕可以觸發三種動作： postback（靜默地向 Bot 發送資料，不顯示在聊天視窗）、 uri（開啟指定 URL，可搭配 LIFF）、 message（以用戶身份傳送指定文字）。 理解這三種 action 型別的差異，是設計良好互動流程的關鍵。

action type: postback
點擊後向 Webhook 發送 postback 事件，data 欄位攜帶自訂字串（最多 300 字元）。 訊息視窗不顯示任何文字，適合傳遞機器可讀的狀態或指令，是最常用的 action 型別。 選填的 displayText 欄位可讓按鈕在聊天室顯示一段人類可讀的文字。
action type: uri
點擊後在 LINE 內建瀏覽器開啟指定 uri。 若 URI 是 https://liff.line.me/{liffId}，則開啟 LIFF 應用程式。 適合連結到外部網站、線上表單、付款頁面。
action type: message
點擊後以用戶身份傳送 text 欄位指定的文字， 內容會顯示在聊天室。行為等同用戶手動輸入，觸發 message Webhook 事件。 適合設計預設回應選項，如「查詢訂單」、「聯絡客服」。
Buttons Template 實作

Buttons Template 是最基本的互動訊息，適合呈現單一情境的選項。 可選擇加入縮圖圖片（thumbnailImageUrl）、標題（title）與說明文字（text）， 最多配置 4 個按鈕動作。

Buttons Template 完整結構
1const buttonsMsg = { 2 type: 'template', 3 altText: '請選擇服務項目', // 不支援 template 的環境顯示此文字 4 contents: { 5 type: 'buttons', 6 thumbnailImageUrl: 'https://cdn.example.com/service-banner.jpg', 7 imageAspectRatio: 'rectangle', // 'rectangle' | 'square' 8 imageSize: 'cover', // 'cover' | 'contain' 9 title: '客服中心', 10 text: '請選擇你需要的服務項目', // 必填，有圖時最多 60 字 11 actions: [ 12 { 13 type: 'postback', 14 label: '查詢訂單', 15 data: 'action=query&type=order', // URLSearchParams 格式建議 16 displayText: '我要查詢訂單' // 可選：聊天室顯示文字 17 }, 18 { 19 type: 'postback', 20 label: '退換貨申請', 21 data: 'action=return' 22 }, 23 { 24 type: 'uri', 25 label: '前往官方網站', 26 uri: 'https://www.example.com' 27 }, 28 { 29 type: 'message', 30 label: '轉接真人客服', 31 text: '我要轉接真人客服' // 以用戶身份傳送 32 } 33 ] 34 } 35}
Postback 資料的解析

postback.data 是一個自訂字串，最多 300 個字元。 建議採用 URL query string 格式（如 action=query&type=order&id=123） 來編碼多個參數，這樣可以用標準的 URLSearchParams 直接解析， 無需自行實作解析邏輯，也大幅降低格式錯誤的風險。

postbackHandler.js — 解析 postback data
1async function postbackHandler(event, client) { 2 // 用 URLSearchParams 解析 data 字串 3 const params = new URLSearchParams(event.postback.data) 4 const action = params.get('action') 5 const type = params.get('type') 6 7 switch (action) { 8 case 'query': 9 return handleQuery(event, client, type) 10 case 'return': 11 return handleReturn(event, client) 12 default: 13 console.warn(`Unknown postback action: ${action}`) 14 return null 15 } 16} 17module.exports = postbackHandler
Confirm Template

Confirm Template 強制限定為兩個按鈕，語意上代表「確認」與「取消」， 適用於刪除資料、送出訂單前的最終確認步驟。 它沒有縮圖和標題欄位，只有必填的 text 和兩個 actions。

Confirm Template — 訂單送出確認
1const confirmMsg = { 2 type: 'template', 3 altText: '確認送出訂單？', 4 contents: { 5 type: 'confirm', 6 text: '確認送出以下訂單？\n商品：限定版手提袋 × 2\n金額：NT$ 1,480', 7 actions: [ 8 { 9 type: 'postback', 10 label: '確認送出', 11 data: 'action=confirm_order&orderId=ORD-2025-0521' 12 }, 13 { 14 type: 'postback', 15 label: '取消', 16 data: 'action=cancel_order&orderId=ORD-2025-0521' 17 } 18 ] 19 } 20}
字數限制 Template Message 各欄位都有嚴格的字數上限。text 欄位在有縮圖圖片時 最多 60 字、無圖時最多 160 字。title 最多 40 字。 label（按鈕文字）最多 20 字。超出限制時 LINE 不會傳送訊息， 而是回傳 400 錯誤，且錯誤訊息不一定明確指出是哪個欄位超出， 排查時要逐一確認。
Lesson 3–4

Carousel Template 與 Image Carousel

Carousel Template 允許在單則訊息中橫向排列最多 10 個卡片（column）， 每個卡片擁有獨立的圖片、標題、說明與按鈕組合。 使用者可以左右滑動瀏覽，非常適合展示商品目錄、服務方案比較、 或多步驟引導中的選項呈現。

Carousel 有一個重要的排版規則：所有欄位的按鈕數量必須一致。 若第一個 column 有 3 個按鈕，其他所有 column 也必須有 3 個按鈕， 否則 LINE API 會回傳錯誤。若某個 column 語意上不需要某個按鈕， 可以放一個帶有佔位 label 的 dummy action 來補齊數量。

Carousel Template — 商品列表（2 欄示範）
1function buildProductCarousel(products) { 2 const columns = products.map(p => ({ 3 thumbnailImageUrl: p.imageUrl, 4 imageBackgroundColor: '#F5F0E8', // 圖片載入前的背景色 5 title: p.name, // 最多 40 字 6 text: `NT$ ${p.price} | ${p.stock > 0 ? '有貨' : '缺貨'}`, 7 actions: [ 8 { 9 type: 'postback', 10 label: '加入購物車', 11 data: `action=addCart&productId=${p.id}` 12 }, 13 { 14 type: 'uri', 15 label: '查看詳情', 16 uri: `https://shop.example.com/products/${p.id}` 17 } 18 ] // 注意：每個 column 的按鈕數必須相同 19 })) 20 21 return { 22 type: 'template', 23 altText: '商品目錄', 24 contents: { 25 type: 'carousel', 26 imageAspectRatio: 'rectangle', // 全 carousel 統一的圖片比例 27 imageSize: 'cover', 28 columns: columns 29 } 30 } 31}
Image Carousel

Image Carousel 是 Carousel 的純圖版本，每個欄位只有一張大圖與一個 action， 沒有文字說明。適合純視覺驅動的場景，如活動橫幅輪播、優惠圖示切換。 最多同樣支援 10 張圖片。

Image Carousel — 活動橫幅輪播

Lesson 3–5

Quick Reply 快速回覆與 Sender 客製化

Quick Reply 是一個可附加到任何訊息型別的補充元件， 傳送後會在用戶的輸入框上方顯示一排可點擊的選項按鈕， 點擊後自動消失並觸發對應的 action。 它與 Buttons Template 的最大差異在於：Quick Reply 是輸入框的延伸， 視覺上屬於「用戶輸入空間」，語意更接近「下一步你想怎麼做？」的引導， 而非獨立的卡片訊息。

Quick Reply 最多支援 13 個選項，每個選項稱為 QuickReplyItem， 包含一個可選的圖示（imageUrl）與一個 action。 action 型別除了 postback、message、uri 之外， 還支援兩個特殊型別：location（開啟位置選擇器）和 camera（開啟相機）。

Quick Reply 附加在文字訊息上
1const msgWithQuickReply = { 2 type: 'text', 3 text: '請選擇你偏好的聯絡方式：', 4 // quickReply 可附加到任意訊息型別，包括 template 5 quickReply: { 6 items: [ 7 { 8 type: 'action', 9 imageUrl: 'https://cdn.example.com/icon-phone.png', // 選填小圖示 10 action: { 11 type: 'postback', 12 label: '電話', 13 data: 'contact=phone' 14 } 15 }, 16 { 17 type: 'action', 18 action: { 19 type: 'postback', 20 label: 'Email', 21 data: 'contact=email' 22 } 23 }, 24 { 25 type: 'action', 26 action: { 27 type: 'location', // 開啟位置選擇器 28 label: '傳送位置' 29 } 30 } 31 ] 32 } 33}
Sender：覆寫 Bot 的顯示名稱與頭像

sender 欄位同樣可附加到任何訊息型別， 讓你在傳送特定訊息時暫時改變 Bot 顯示的名稱與頭像。 這在多角色 Bot（例如「客服小助」負責問答、「訂單系統」負責通知） 或需要區分人工客服介入的情境中很有用。 覆寫只對該則訊息有效，不影響 Bot 的全域設定。

使用 sender 暫時改變 Bot 顯示身份
1const notificationMsg = { 2 type: 'text', 3 text: '您的訂單 #ORD-2025-0521 已確認出貨，預計明日送達。', 4 // 傳送此訊息時，Bot 顯示為「訂單通知系統」 5 sender: { 6 name: '訂單通知系統', // 最多 20 字元 7 iconUrl: 'https://cdn.example.com/order-bot-icon.png' // HTTPS，1MB 以內 8 } 9}
建立訊息建構器（Message Builder Pattern）

隨著訊息型別增多，直接在 handler 中拼湊 JSON 會讓程式碼難以閱讀。 建立專屬的 builders/messageBuilders.js 模組， 將每種訊息型別封裝成具名函式，讓 handler 的程式碼聚焦在業務邏輯， 而非 JSON 結構的細節。

src/builders/messageBuilders.js — 訊息工廠函式
1// 文字訊息工廠 2const text = (message) => ({ type: 'text', text: message }) 3 4// 圖片訊息工廠 5const image = (url, preview) => ({ 6 type: 'image', originalContentUrl: url, previewImageUrl: preview || url 7}) 8 9// Confirm Template 工廠 10const confirm = ({ text: bodyText, yesLabel, yesData, noLabel, noData }) => ({ 11 type: 'template', 12 altText: bodyText.slice(0, 40), 13 contents: { 14 type: 'confirm', text: bodyText, 15 actions: [ 16 { type: 'postback', label: yesLabel, data: yesData }, 17 { type: 'postback', label: noLabel, data: noData } 18 ] 19 } 20}) 21 22// 帶 Quick Reply 的文字訊息工廠 23const textWithQR = (message, options) => ({ 24 type: 'text', text: message, 25 quickReply: { 26 items: options.map(opt => ({ 27 type: 'action', 28 action: { type: 'postback', label: opt.label, data: opt.data } 29 })) 30 } 31}) 32 33// 使用範例：handler 中只需呼叫工廠函式 34// client.replyMessage(token, [ 35// builders.text('請問你想查詢什麼？'), 36// builders.textWithQR('選擇查詢類別', [ 37// { label: '訂單狀態', data: 'query=order' }, 38// { label: '點數查詢', data: 'query=points' } 39// ]) 40// ]) 41 42module.exports = { text, image, confirm, textWithQR, buildProductCarousel }

Message Builder Pattern 讓訊息建構的邏輯集中在一個地方， 日後若 LINE API 規格調整（例如某欄位改名）， 只需修改對應的 builder 函式，不需要在整個程式碼庫搜尋所有使用處。 這也讓單元測試更容易撰寫——你可以直接測試 builder 函式的輸出是否符合預期結構， 而不需要模擬整個 LINE 環境。

架構建議 本章完成後，你的 src/ 目錄應新增 builders/messageBuilders.js， 並將 handlers/ 中的各種 handler 改為呼叫 builder 函式， 而非直接構造 JSON 物件。這個小小的重構在後期維護時將節省大量時間。 同時建議為 builders 撰寫幾個 jest 測試， 驗證關鍵欄位的字數限制守衛（例如 text 超過 160 字時自動截斷或拋出錯誤）。
下一章 Flex Message 是 Template Message 的進化版， 能做到幾乎所有 Template Message 可以做的事，並大幅超越其排版限制。 但 Template Message 的建構語法更簡單，適合快速迭代的早期階段； Flex Message 則需要更多設計工作，適合成熟後的品質打磨。 兩者並不互斥，同一個 Bot 可以混用。

postback 的選填欄位，點擊後顯示於聊天室的可讀文字。
Quick Reply
可附加於任何訊息的輸入框選項，最多 13 個，點後消失。
Multicast
批次推送給最多 500 個 userId，計費 API。
sender
單則訊息的顯示名稱與頭像覆寫，不影響全域設定。
