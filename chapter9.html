Lesson 9–1

部署平台選型與 Docker 容器化

LINE Bot 必須部署在能接收外部 HTTPS 請求的公開主機上—— 本地的 localhost 只適合開發期間搭配 ngrok 使用。 正式環境的平台選擇影響可用性、擴展能力和運維複雜度， 沒有絕對最佳答案，只有最適合當下規模的選擇。

主要部署平台比較
平台定價模式適用規模主要限制
Railway 用量計費，有免費額度 個人專案、小型商用 免費方案每月有休眠機制
Render 免費方案 + 付費 小至中型 免費方案 15 分鐘無流量自動休眠，首次請求延遲高
Fly.io 用量計費，有免費額度 中型，需要多地區 設定相對複雜，需 flyctl CLI
Google Cloud Run 按請求計費，冷啟動 流量不穩定的場景 冷啟動延遲可能影響 Webhook 回應時間
AWS EC2 / GCP VM 固定月費 大型，需完整控制 需自行管理作業系統、安全更新
Heroku 付費方案起 小至中型 已無免費方案，成本較高
LINE Webhook 的回應時間限制 LINE 平台要求 Webhook 端點必須在 1 秒內回應 HTTP 200， 否則視為失敗並重試。若使用會冷啟動的平台（Cloud Run、Render 免費方案）， 冷啟動時間可能達 3–10 秒，導致 LINE 誤判失敗。 解法：使用最小實例數設定（min instances = 1）保持容器常駐， 或先回應 200 再非同步處理業務邏輯。

Lesson 9–2

環境變數管理與 GitHub Actions CI/CD

生產環境的密鑰（API Key、資料庫密碼、Channel Secret）絕對不能放進 Git 倉庫。 正確做法是透過平台的環境變數設定介面注入， 本地開發則用 .env 檔案搭配 dotenv 載入， 並確保 .env 列在 .gitignore 中。 CI/CD 流水線則使用 GitHub Secrets 存放這些敏感值， 在 Actions workflow 執行時動態注入。

部署 Pipeline 的四個階段
CI/CD 流水線
1
git push → 觸發 Actions推送到 main 分支自動啟動 workflow
↓
2
安裝依賴 + 執行測試npm ci → npm test，失敗即停止，不部署
↓
3
建置 Docker 映像並推送docker build → push to registry（或直接 Railway 部署）
↓
4
部署完成 + 健康檢查新版本上線，執行 /health 確認服務正常，異常時回滾
資料庫 Migration 的時序問題 若應用程式啟動時自動執行 migration（knex migrate:latest）， 在多實例部署時可能多個實例同時執行 migration，造成衝突。 正確做法是將 migration 作為 CI/CD 的獨立步驟， 在新版本容器啟動之前執行，確保只跑一次。

console.log('用戶傳了訊息') 在開發時夠用， 但在生產環境中幾乎沒有診斷價值。 當 Bot 在凌晨三點出現問題，你需要的是能快速篩選、搜尋、 告警的結構化日誌系統——每條日誌都是 JSON， 包含時間戳、嚴重程度、userId、事件類型等可查詢的欄位。

Lesson 9–4

健康檢查、指標收集與告警

知道服務「壞了」通常比知道服務「快要壞了」晚了很久。 健康檢查端點讓部署平台和監控系統能定期確認服務狀態； 指標收集讓你看到 Bot 的處理量、回應時間趨勢； 告警讓問題在影響用戶之前就通知到你。

使用 Betteruptime 做外部監控

內部健康檢查解決不了「整個伺服器掛掉」的場景—— 伺服器都不跑了，自然也無法檢查自己。 外部監控服務（Betteruptime、UptimeRobot、Pingdom）從外部定期打你的 /health 端點， 服務無回應超過設定閾值時立刻發送告警（Email、SMS、LINE Notify、PagerDuty）。 每隔 1 分鐘打一次的免費方案對大多數 Bot 場景已足夠。

Lesson 9–5

Webhook 安全性與生產加固清單

公開在網際網路上的 Webhook 端點， 任何人都能對它發送 POST 請求。 沒有驗證機制的端點等於讓任何人都能觸發你的 Bot 邏輯、 消耗你的 OpenAI 配額、甚至偽造事件操縱業務資料。 LINE SDK 的 middleware 提供簽名驗證； 除此之外還有一系列加固措施共同構成防禦縱深。

LINE Webhook 簽名驗證原理

LINE 平台在每次請求的 X-Line-Signature 標頭中， 附上以 Channel Secret 作為 HMAC-SHA256 密鑰對請求 body 計算的簽名。 後端以相同方式計算並對比，若不符則拒絕請求。 攻擊者不知道你的 Channel Secret，無法偽造合法的簽名。 @line/bot-sdk 的 middleware 自動完成這個驗證，但有一個關鍵前提： body 必須是原始位元組（raw buffer），不能是已解析的 JSON 物件。

index.js — 正確的 body parser 設定順序（關鍵）
1const express = require('express')
2const line = require('@line/bot-sdk')
3const app = express()
4

6// line.middleware 會保留 raw body 用於簽名驗證
7app.post(
8'/webhook',

10webhookRouter
11)
12
13// 其他 API 路由：使用標準 JSON body parser
14app.use(express.json())
15app.use('/api', apiRouter)

手動驗證 LINE 簽名（不使用 SDK middleware 時）
手動 HMAC-SHA256 簽名驗證
1const crypto = require('crypto')
2
3function verifyLineSignature(rawBody, signature, channelSecret) {
4const expected = crypto
5.createHmac('sha256', channelSecret)
6.update(rawBody) // rawBody 必須是 Buffer 或原始字串
7.digest('base64')
8
9// timingSafeEqual 防止時序攻擊（timing attack）
10return crypto.timingSafeEqual(
11Buffer.from(expected),
12Buffer.from(signature),
13)
14}

項目做法優先級

開發環境與生產環境的 Channel 隔離 開發期間應使用獨立的測試用 LINE Channel，與生產 Channel 完全分開。 這樣做有兩個好處：一，開發時的測試訊息不會影響真實用戶； 二，即使開發環境的 Channel Secret 洩露，生產環境不受影響。 GitHub Actions 也應設定兩組 secrets：PROD_* 和 STAGING_*， 不同分支使用不同環境的密鑰部署。

LINE Webhook 簽名驗證為何要求 body 必須是 raw buffer，而不能是已解析的 JSON？
A.JSON 格式的 body 無法轉換為 base64，所以無法計算 HMAC
B.HMAC 對位元組序列計算，JSON.parse 再 stringify 可能改變欄位順序或空白，導致簽名不符
C.LINE 的 SDK 有 bug，只有 raw buffer 才不會崩潰
D.已解析的 JSON 物件佔用記憶體過多，驗證時會出現 out-of-memory 錯誤
正確。HMAC-SHA256 是對原始位元組序列計算的。LINE 傳送的簽名是針對 原始 HTTP body 字串計算的。若你先用 express.json() 解析， 再用 JSON.stringify() 轉回字串，可能因為 key 排序差異、多餘空白、 或 Unicode 正規化等因素，產生與原始 body 不完全相同的字串，導致簽名驗證失敗。 正確做法是在 Webhook 路由上讓 line.middleware() 攔截原始 body， 其餘路由再使用 express.json()。
