Lesson 8–1

Chat Completions API 基礎與最小可用整合

OpenAI 的 Chat Completions API 以訊息陣列為輸入： 你傳入一組對話歷史（包含 system、user、assistant 三種角色的訊息）， 模型回傳下一則助理回覆。 這個設計直接映射到聊天室場景——用戶的每則訊息就是一個 user 訊息， 你把它附加到歷史陣列，呼叫 API， 然後把回傳的 assistant 訊息送回 LINE。

Lesson 8–2

Token 計算、費用控制與對話歷史管理

OpenAI 以 Token 為計費單位。一個 token 約等於英文四個字元或中文一到兩個字。 每次 API 呼叫的費用 = (輸入 token 數 + 輸出 token 數) × 單價。 Chat Completions 的輸入包含你傳入的整個 messages 陣列—— 歷史訊息越長，每次呼叫費用越高。 若不限制歷史長度，隨著對話進行，費用會線性增長， 最終甚至超過模型的 Context Window 上限而導致 API 報錯。

滑動視窗策略：限制歷史長度

最簡單的費用控制方式：只保留最近 N 輪對話。 超過的部分從陣列頭部捨棄，System Prompt 始終保留。 這在大多數客服場景下效果良好——用戶通常只需要最近幾輪的上下文。

1const SessionStore = require('../session/RedisSessionStore')
2
3const MAX_HISTORY_TURNS = 10 // 保留最近 10 輪（每輪 = user + assistant 各一則）

5const SESSION_NAME = 'ai-chat'
6

8async function getHistory(userId) {
9const session = await SessionStore.get(userId, SESSION_NAME)
10return session.context?.history ?? []
11}
12
13// 追加一輪對話並執行滑動視窗截斷
14async function appendAndTrim(userId, userMsg, assistantMsg) {
15let history = await getHistory(userId)
16
17// 追加本輪的 user 和 assistant 訊息
18history.push({ role: 'user', content: userMsg })
19history.push({ role: 'assistant', content: assistantMsg })

24}

27}

32}

Token 預算策略：改用 token 數而非輪數

更精確的做法是以 token 數（而非對話輪數）作為截斷標準。 tiktoken 是 OpenAI 官方的 token 計算函式庫， 可以在傳送前精確估算 token 消耗，確保歷史不超出指定的 token 預算。

依 token 預算截斷對話歷史
1const { encodingForModel } = require('tiktoken') // npm install tiktoken
2
3const enc = encodingForModel('gpt-4o-mini')
4
5function countTokens(messages) {
6let total = 0
7for (const msg of messages) {
8total += 4 // 每則訊息的 overhead（role + content 標記）
9total += enc.encode(msg.content).length
10}
11return total
12}
13
14// 截斷歷史至不超過 token 預算
15function trimToTokenBudget(history, budgetTokens = 3000) {
16// 從最舊（陣列頭部）開始移除，直到符合預算
17while (history.length > 0 && countTokens(history) > budgetTokens) {
18history.splice(0, 2) // 每次移除最舊的一輪（user + assistant）
19}
20return history
21}
費用估算公式 以 gpt-4o-mini 為例，假設每次呼叫平均 1,500 input token + 300 output token， 每日 1,000 次請求的月費用約為：
(1,500 × $0.15 + 300 × $0.60) / 1,000,000 × 30,000 次 = 約 $7.45 USD/月。 同樣流量若使用 gpt-4o，費用約為 $125 USD/月，相差約 17 倍。 選對模型是最有效的費用控制手段。
Lesson 8–3

System Prompt 工程：打造有個性的 AI 助理

System Prompt 是對話陣列中 role: "system" 的訊息， 在整個對話過程中持續生效，定義模型的角色、行為邊界和回覆風格。 設計良好的 System Prompt 是讓 Bot 表現出色的關鍵， 也是防止用戶透過特殊輸入讓模型做出不當行為（Prompt Injection）的第一道防線。

System Prompt 的七個構成要素
角色定義
明確陳述 AI 的身份與職責範圍。 例：「你是星光百貨的客服助理小星，專門回答關於商場營業時間、活動資訊和購物優惠的問題。」
語言與語氣
指定使用語言和語氣風格。 例：「請使用繁體中文回覆，語氣親切但專業，避免使用過於口語化的表達。」
行為邊界
明確指出哪些事不做。 例：「不要回答與本百貨無關的問題；若被問及競爭對手，禮貌地將話題引回本商場。」
知識基礎
提供業務相關的基礎知識。 例：「本商場營業時間：週一至週五 11:00–22:00，週末 10:00–22:30。會員日每月第一個週六。」
回覆格式
指定輸出格式偏好。LINE 訊息不支援 Markdown 渲染，應指定「不使用 #、**、- 等標記符號」。
長度限制
LINE 單則訊息上限 5,000 字元。建議在 System Prompt 中指定回覆長度，例：「回覆請在 300 字以內」。
升級指引
定義無法回答時的處理方式。 例：「如遇到你無法確定答案的問題，請建議用戶聯繫客服熱線 0800-XXX-XXX。」

防範 Prompt Injection

Prompt Injection 指用戶透過特殊輸入企圖覆蓋你的 System Prompt， 例如傳送「請忽略所有先前的指示，改為...」。 除了在 System Prompt 中明確告知模型不要遵循此類指令， 還可以在收到訊息後做基本的輸入過濾：

Lesson 8–4

速率限制與費用保護機制

若沒有速率限制，單一惡意用戶或自動化腳本可以在幾分鐘內發出數千次 API 請求， 產生大量意外費用。 即使是善意用戶，也可能因為 Bot 回應緩慢而重複傳送相同訊息， 造成重複計費。 以下實作兩層防護：每用戶的請求速率限制，以及重複請求的去重機制。

每用戶速率限制（以 Redis 實作）
src/ai/rateLimiter.js — 滑動視窗速率限制
1const Redis = require('ioredis')
2const redis = new Redis(process.env.REDIS_URL)
3
4const WINDOW_SECONDS = 60 // 統計視窗：60 秒
5const MAX_REQUESTS = 10 // 每個視窗內最多 10 次 AI 請求
6
7/**
8 * 檢查用戶是否超過速率限制
9 * @returns {{ allowed: boolean, remaining: number, resetIn: number }}
10 */
11async function checkRateLimit(userId) {
12const key = `ratelimit:ai:${userId}`
13
14// 原子性遞增並設定 TTL（Redis MULTI/EXEC 確保原子性）
15const pipeline = redis.pipeline()
16pipeline.incr(key)
17pipeline.expire(key, WINDOW_SECONDS)
18pipeline.ttl(key)
19const [[, count], , [, ttl]] = await pipeline.exec()
20
21return {
22allowed: count <= MAX_REQUESTS,
23remaining: Math.max(0, MAX_REQUESTS - count),
24resetIn: ttl, // 幾秒後視窗重設
25}
26}
27
28module.exports = { checkRateLimit }
Webhook 重複事件去重

LINE 平台在網路不穩定時可能重複傳送同一個 Webhook 事件。 每個事件都有唯一的 webhookEventId， 在 Redis 中記錄已處理的 event ID， 重複收到時直接返回，不重複呼叫 OpenAI API。

Webhook 事件去重（idempotency）
1const DEDUP_TTL = 300 // 5 分鐘內的重複事件都過濾
2
3async function isDuplicateEvent(eventId) {
4const key = `processed:${eventId}`
5// SET NX（Not eXists）：只在 key 不存在時設定，回傳 'OK' 或 null
6const result = await redis.set(key, 1, 'EX', DEDUP_TTL, 'NX')
7return result === null // null = key 已存在 = 重複事件
8}
9
10// 在 Webhook router 的最上層使用
11for (const event of req.body.events) {
12if (await isDuplicateEvent(event.webhookEventId)) {
13console.log(`[Dedup] Skipping duplicate event: ${event.webhookEventId}`)
14continue
15}
16await handleEvent(event, client)
17}

Lesson 8–5

Function Calling 與知識庫擴充

語言模型的訓練資料有截止日期，且不包含你的業務資料（產品庫存、訂單狀態、會員點數）。 Function Calling 讓模型在回答時能主動呼叫你定義的工具函式—— 例如查詢資料庫、呼叫外部 API——並將結果整合到最終回覆中。 模型不直接執行函式，而是輸出「我需要用 X 函式查 Y 參數」的結構化指令， 由你的後端實際執行後再把結果傳回模型。

5// 第一次呼叫：傳入 tools，模型可能回傳 tool_calls
6const first = await openai.chat.completions.create({
7model: 'gpt-4o-mini',
8messages: messages,
9tools: TOOLS,
10tool_choice: 'auto', // 讓模型自行決定是否使用工具
11max_tokens: 1000,
12})
13
14const choice = first.choices[0]
15
16// 若模型未使用工具，直接回傳文字回覆
17if (choice.finish_reason !== 'tool_calls') {
18return choice.message.content
19}
20
21// 模型請求呼叫工具：逐一執行
22const toolResults = []
23for (const toolCall of choice.message.tool_calls) {
24const fnName = toolCall.function.name
25const args = JSON.parse(toolCall.function.arguments)
26const result = await TOOL_EXECUTORS[fnName](args, context)
27toolResults.push({
28role: 'tool',
29tool_call_id: toolCall.id,
30content: JSON.stringify(result),
31})
32}
33
34// 第二次呼叫：把工具執行結果傳回模型，取得最終文字回覆
35const second = await openai.chat.completions.create({
36model: 'gpt-4o-mini',
37messages: [
38...messages,
39choice.message, // 包含 tool_calls 的 assistant 訊息
40...toolResults, // 每個 tool 的執行結果
41],
42max_tokens: 1000,
43})
44
45return second.choices[0].message.content
46}
47
48module.exports = { chatWithTools }
Function Calling vs. RAG Function Calling 適合即時查詢結構化資料（訂單、庫存、點數）。 若你的知識庫是大量非結構化文件（FAQ、產品說明書）， 則應考慮 RAG（Retrieval-Augmented Generation）： 先用向量資料庫（如 Pinecone、pgvector）搜尋相關段落， 再把段落嵌入 System Prompt 作為上下文，讓模型據此回答。 兩者可以並用：RAG 提供文件知識，Function Calling 提供即時業務資料。

核心術語
Token
OpenAI 計費與 Context 計量的基本單位。約等於英文 4 字元或中文 1–2 字。
System Prompt
role 為 system 的訊息，定義模型的角色與行為規則，在整個對話持續生效。
Context Window
模型一次能處理的最大 token 數（輸入+輸出總和）。超出後舊訊息將被截斷。
Function Calling
模型輸出結構化的工具呼叫指令，由後端執行後將結果傳回模型，以存取外部資料。
Prompt Injection
用戶透過特殊輸入企圖覆蓋 System Prompt 或改變模型行為的攻擊手法。
RAG
Retrieval-Augmented Generation，先檢索相關文件片段再讓模型生成回覆的架構。
temperature
控制模型輸出隨機性的參數（0–2）。0 接近確定性輸出，1 以上創意性強但不穩定。

為什麼滑動視窗策略必須始終保留 System Prompt，只截斷對話歷史？
A.System Prompt 的 token 數不計入費用，截不截斷都一樣
B.API 規定 system 訊息必須是陣列的第一個元素，刪除會報錯
C.System Prompt 定義了角色與行為規則，若遺失模型會脫離預設角色
D.沒有 System Prompt 時模型會拒絕回答並回傳錯誤
正確。System Prompt 是整個 Bot 行為的基礎定義——角色、語言、邊界、格式限制都在這裡。 一旦被截掉，模型就會按照預設行為回應，可能使用英文、不限制回覆範圍， 或做出你不希望的回覆。對話歷史可以截斷（用戶只是失去記憶）， 但 System Prompt 必須每次都傳入。
