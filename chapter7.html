LIFF
應用程式開發

LIFF（LINE Front-end Framework）讓你在 LINE 的聊天介面內嵌入 完整的網頁應用程式。用戶不需要離開 LINE、不需要另行登入， 就能使用表單、地圖、選單等豐富的互動介面，並在操作完成後直接傳送訊息回聊天室。 本章從 LIFF 的運作原理到實作帳號綁定流程，系統性地建立 Bot 與網頁之間的雙向溝通能力。

學習目標
  • 理解 LIFF 的執行環境與 context 差異
  • 建立並設定 LIFF App，取得 LIFF ID
  • 用 liff.init() 完成初始化並取得用戶 ID Token
  • 在後端驗證 ID Token，安全地識別用戶身份
  • 實作 LINE 帳號與業務系統的綁定流程
  • 使用 liff.sendMessages() 從 LIFF 傳送訊息回聊天室
Lesson 7–1

LIFF 的執行環境與架構概念

LIFF 是一個在 LINE 客戶端內部開啟的 WebView 環境。 從技術角度看，它就是一個普通的網頁； 從用戶體驗角度看，它是嵌入在聊天室中的應用程式介面， 開啟與關閉都不中斷對話。 LIFF 的最大價值在於：網頁可以透過 LIFF SDK 存取 LINE 平台的原生能力—— 取得已登入用戶的身份、直接傳送訊息到聊天室、存取裝置的掃碼功能等。

三種 LIFF 視窗尺寸
尺寸視窗高度適合場景
compact 螢幕高度 50% 簡短的確認表單、快速選擇、QR Code 掃描
tall 螢幕高度 80% 較長的表單、商品列表、地址輸入
full 螢幕高度 100% 複雜的多步驟流程、地圖、內嵌購物頁面
LIFF Context 的差異

LIFF 可以從兩種管道開啟：從 LINE 內部（InApp）或從外部瀏覽器（External）。 這個差異影響能使用的 API 範圍。 liff.getContext() 回傳的 type 欄位告訴你當前執行環境：

utou
從一對一聊天開啟。liff.sendMessages() 可用，訊息發送到該對話。
room
從多人房間開啟。sendMessages() 可用，訊息發送到該房間。
group
從群組開啟。sendMessages() 可用，訊息發送到該群組。
external
從外部瀏覽器開啟。sendMessages() 不可用，其他 API 仍可使用。
none
透過 LINE 以外的管道開啟，且未登入。部分 API 需要先呼叫 liff.login()。

LIFF 前端與 Bot 後端是兩個獨立的部署單元。 LIFF 頁面可以托管在任何支援 HTTPS 的靜態主機上， Bot 伺服器則負責接收 LINE 的 Webhook 和來自 LIFF 頁面的 API 請求。 兩者透過你自己的後端 API 溝通，LIFF SDK 負責橋接 LINE 的身份系統。

Lesson 7–2

建立 LIFF App 與初始化 SDK

在 LINE Developers Console 建立 LIFF App 後， 系統會分配一個 LIFF ID（格式：xxxxxxxxxx-yyyyyyyy）。 LIFF URL 的格式為 https://liff.line.me/{liffId}， 這個 URL 可以放在 Bot 訊息的按鈕、Rich Menu 的熱區、或 Quick Reply 中， 讓用戶點擊後直接開啟你的 LIFF 頁面。

在 Console 設定 LIFF App
Endpoint URL
LIFF 頁面的 HTTPS 網址。開發時可用 ngrok 產生暫時的 HTTPS URL， 生產環境需使用正式域名。必須是 HTTPS，不接受 HTTP。
Size
視窗尺寸：compact、tall、full。建立後可修改。
Scope
LIFF 需要存取的用戶資料範圍。 profile：取得顯示名稱與大頭貼； openid：取得 ID Token（帳號綁定必須勾選）； email：取得電子郵件（需用戶額外同意）。
Bot link feature
設為 Aggressive 時，用戶開啟 LIFF 頁面若尚未加好友， 會自動彈出加好友的提示視窗。

Lesson 7–3

ID Token 驗證與安全身份識別

LIFF SDK 的 liff.getIDToken() 回傳一個 JWT 格式的 ID Token。 這個 token 是由 LINE 平台簽發的，包含用戶的 sub（即 LINE userId） 和其他宣告（claims），並以 LINE 的私鑰簽名。 後端向 LINE 的驗證端點送出這個 token， 確認它確實是 LINE 簽發的、且對應到你的 Channel， 才能安全地從中取出用戶身份。

18}
後端 — 驗證 ID Token 並取得 userId（Node.js）
1const express = require('express')
2const axios = require('axios')
3const router = express.Router()

8/**
9 * 向 LINE 驗證 ID Token，回傳 payload（含 sub = lineUserId）
10 * 驗證失敗（偽造 / 過期 / 錯誤 Channel）會丟出例外

12async function verifyIdToken(idToken) {
13const params = new URLSearchParams({
14id_token: idToken,
15client_id: process.env.LINE_CHANNEL_ID, // Channel ID（非 Access Token）
16})
17
18const { data } = await axios.post(LINE_VERIFY_URL, params, {
19headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
20})
21
22// data.sub 是該 Channel 下唯一識別用戶的 LINE userId
23// data.name 是顯示名稱，data.picture 是大頭貼 URL
24return data
25}
26
27router.post('/api/liff/bind', async (req, res) => {
28const { idToken, memberCode } = req.body
29
30if (!idToken || !memberCode) {
31return res.status(400).json({ error: 'idToken and memberCode are required' })
32}
33
34try {
35// Step 1: 向 LINE 驗證 ID Token，取得可信任的 userId
36const payload = await verifyIdToken(idToken)
37const lineUserId = payload.sub
38
39// Step 2: 在資料庫中查找該 memberCode 對應的業務帳號
40const member = await MemberRepo.findByCode(memberCode)
41if (!member) {
42// 找不到對應會員，回傳錯誤
43return res.status(404).json({ error: 'Member not found' })
44}
45

47await MemberRepo.update(member.id, { lineUserId, boundAt: new Date() })
48
49// Step 4: 切換 Rich Menu（從訪客版→會員版）
50await syncRichMenu(lineUserId, { tier: member.tier }, lineClient)
51
52return res.json({ success: true, displayName: payload.name })
53} catch (err) {
54console.error('Bind error:', err.message)
55return res.status(401).json({ error: 'ID Token verification failed' })
56}
57})
client_id 與 Channel Access Token 的區別 驗證 ID Token 時傳入的是 client_id，對應的是 Channel 的 Channel ID（純數字）， 而不是 Channel Access Token（長字串）。 Channel ID 可以在 LINE Developers Console 的 Basic settings 頁面找到。 兩者混淆是最常見的初始化錯誤之一，請確認 .env 中的 LINE_CHANNEL_ID 存放的是 Channel ID 而非 Access Token。
Lesson 7–4

帳號綁定流程完整實作

帳號綁定是 LIFF 最核心的應用場景：讓用戶在 LINE 聊天室中， 用他們在實體店或官網的既有帳號登入， 將 LINE 身份與業務系統的會員身份永久關聯。 綁定完成後，Bot 收到任何來自該用戶的 Webhook 事件， 就能直接從資料庫查到他的完整業務資料，無需再次詢問。

liff.sendMessages() 的限制 liff.sendMessages() 只能在 LINE 內部開啟的 LIFF 中使用（context type 為 utou / room / group）。 從外部瀏覽器開啟的 LIFF（type 為 external）呼叫此方法會拋出例外。 使用前務必先檢查 liff.isInClient()，或在 console 設定時要求從 LINE 內開啟。

本地開發設定：Mock LIFF 與 ngrok

本地開發 LIFF 有兩個主要挑戰： 第一，LIFF 必須透過 HTTPS 存取，本地的 localhost:3000 不符合； 第二，若不在 LINE 客戶端內執行，liff.init() 無法取得真實的 LINE 身份。 以下兩種解法分別對應不同的開發需求：

方案 A — 用 ngrok 暴露本地 HTTPS（最接近真實環境）

為何帳號綁定時不能直接信任前端傳來的 userId，而必須使用 ID Token 驗證？
A.ID Token 的傳輸速度比 userId 更快
B.前端的 userId 可被偽造，ID Token 由 LINE 簽名，後端能驗證真偽
C.liff.getProfile() 在外部瀏覽器無法使用，只有 ID Token 可用
D.LINE 平台強制要求必須使用 ID Token，不使用會導致 API 請求失敗
正確。任何人都可以在 HTTP 請求中偽造一個 userId 字串。 ID Token 是由 LINE 以私鑰簽名的 JWT，後端向 LINE 驗證端點送出後， LINE 會確認簽名有效且 client_id 符合你的 Channel， 才回傳可信任的 sub（userId）。這個驗證步驟是整個安全模型的基礎，不能省略。
