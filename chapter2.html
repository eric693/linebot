Webhook 事件處理
與訊息型別

第一章的 Echo Bot 只處理了最簡單的文字訊息。真實世界的 Bot 必須因應數十種事件類型——追蹤、取消追蹤、加入群組、圖片、 位置資訊、按鈕點擊——並為每種情境提供正確的回應。 本章建立一套可擴展的事件路由架構，讓你的程式碼在功能增長後仍然易於維護。

學習目標
  • 掌握所有 LINE Webhook 事件型別與其資料結構
  • 建立模組化的 EventRouter 分離各類事件邏輯
  • 實作 follow 歡迎訊息並取得使用者 Profile
  • 正確處理 unfollow 等無 replyToken 的事件
  • 下載並儲存圖片、影片等媒體型訊息
  • 理解 source 來源欄位在群組情境的差異
Lesson 2–1

Webhook 事件型別完整分類

LINE 的 Webhook 事件不只有訊息。每當使用者與你的 Bot 或群組發生任何互動—— 無論是加入好友、傳送貼圖、點選按鈕、分享位置，還是機器人被踢出群組—— LINE 都會向你的 Webhook URL 發送一個對應的事件物件。 理解這些事件的結構與差異，是建立任何實用 Bot 的前提。

所有事件都封裝在同一個 Webhook payload 結構中，差別只在 event.type 欄位的值，以及該型別額外攜帶的資料欄位。以下是完整的事件分類表：

event.type 觸發時機 含 replyToken 主要用途
message 使用者傳送任何類型的訊息 有 核心互動，處理文字、圖片、貼圖等
follow 使用者加入好友或解除封鎖 有 發送歡迎訊息、記錄新用戶
unfollow 使用者封鎖 Bot 無 記錄用戶流失、清理資料
join Bot 被加入群組或多人聊天室 有 傳送群組歡迎訊息、說明指令
leave Bot 被移除出群組 無 記錄群組離開事件
postback 使用者點選含 postback action 的按鈕 有 處理選單選項、多步驟流程
memberJoined 新成員加入 Bot 所在的群組 有 歡迎新群組成員
memberLeft 成員離開 Bot 所在的群組 無 記錄群組成員變動
beacon 使用者進入 LINE Beacon 裝置範圍 有 實體場域的到場觸發
accountLink 用戶完成帳號連結流程 有 LINE Login 帳號綁定確認

注意 unfollow、leave、memberLeft 三個事件都沒有 replyToken——因為這些事件代表使用者主動脫離關係， 在這個時間點你已無法向他們傳送任何訊息。唯一能做的事是記錄到資料庫。 若你錯誤地對這些事件呼叫 replyMessage()， LINE API 會回傳 400 錯誤，導致 Bot 拋出未捕捉的 Promise 例外。

事件來源（source）欄位

每個事件的 source 欄位描述了事件的發生情境，共有三種型別：

source.type: "user"
一對一私訊聊天。source.userId 即為使用者的唯一識別碼， 此 ID 在同一個 Provider 下的所有 Channel 中保持一致， 可跨 Channel 識別同一個用戶。
source.type: "group"
Bot 所在的群組。同時提供 source.groupId 與 source.userId， 但注意：userId 不一定存在（部分群組隱私設定下不提供）， 取用前必須先檢查。
source.type: "room"
多人聊天室（非正式群組）。提供 source.roomId， 但 userId 同樣不一定存在。 現行 LINE 版本較少使用 room，主流為 group。
實作要點 取用 source.userId 之前，必須先確認 source.type === 'user' 或使用可選鏈寫法 event.source?.userId，並加上存在性檢查。 在群組情境下直接存取 source.userId 可能得到 undefined， 若你將其用作資料庫的 key，會產生難以追蹤的 bug。
Lesson 2–2

訊息子型別：message event 的內部結構

event.type === 'message' 只是最外層的分類， 內部還有 event.message.type 這第二層型別， 決定訊息的實際內容。LINE 支援的訊息子型別共有七種：

message.type 關鍵欄位 說明
text message.text 純文字。若訊息包含 LINE emoji，會以 Unicode 字元夾帶，可直接讀取。
image message.id 圖片。不含圖片資料，需用 getMessageContent(id) 另行下載二進位流。
video message.id, message.duration 影片。同樣需另行下載，duration 為毫秒。
audio message.id, message.duration 語音訊息。格式為 M4A，可導入語音識別（STT）API。
file message.fileName, message.fileSize 檔案傳送。同樣需另行下載，注意 LINE 有檔案大小上限。
location message.latitude, message.longitude, message.address 位置資訊。直接包含座標與地址，無需另行請求。
sticker message.packageId, message.stickerId LINE 貼圖。可根據 packageId 判斷貼圖包，進行趣味互動回應。

image、video、audio、file 這四種型別的訊息事件本身都只攜帶 message.id， 不包含實際的二進位內容。你需要在取得事件後， 另外呼叫 client.getMessageContent(messageId) 取得一個可讀的 Node.js Readable stream，再將其寫入磁碟或傳入處理流程。 這個下載動作有時間限制，LINE 不會無限期保存用戶上傳的媒體檔案。

Lesson 2–3

建立可擴展的 EventRouter 架構

第一章的 index.js 把所有邏輯塞在一個 handleEvent 函式中， 當事件型別增加到五個、十個時，這個函式會變得難以閱讀和測試。 正確的做法是建立一個路由層，讓每種事件有各自專屬的處理模組。 這個架構與 Express 的 Router 概念如出一轍。

檔案結構規劃
建議的模組化目錄結構
1src/
2├── router.js // 主路由，依 event.type 分派
3├── handlers/
4│ ├── messageHandler.js // 處理所有 message 事件
5│ ├── followHandler.js // 處理 follow / unfollow
6│ ├── postbackHandler.js // 處理 postback 按鈕事件
7│ └── groupHandler.js // 處理 join / leave / member 事件
8└── services/
9└── userService.js // 使用者資料存取（下一章加入）
主路由：router.js

路由層的職責只有一個：根據 event.type 將事件交給正確的 handler， 自身不包含任何業務邏輯。使用物件 dispatch table（而非 switch） 的好處是可以在執行期動態注入或替換 handler，更便於單元測試。

src/router.js — 主事件路由
1const messageHandler = require('./handlers/messageHandler')
2const followHandler = require('./handlers/followHandler')
3const postbackHandler = require('./handlers/postbackHandler')
4const groupHandler = require('./handlers/groupHandler')
5
6// 事件型別 → 處理函式的對照表
7const HANDLERS = {
8message: messageHandler,
9follow: followHandler.onFollow,
10unfollow: followHandler.onUnfollow,
11postback: postbackHandler,
12join: groupHandler.onJoin,
13leave: groupHandler.onLeave,
14memberJoined: groupHandler.onMemberJoined
15}
16
17/**
18* 主路由函式，由 index.js 的 Webhook handler 呼叫
19* @param {Object} event LINE Webhook 事件物件
20* @param {Object} client LINE SDK Client 實例
21*/
22async function router(event, client) {
23const handler = HANDLERS[event.type]
24
25// 沒有對應 handler 的事件型別：靜默忽略，回傳 null
26if (!handler) {
27console.log(`[Router] Unhandled event type: ${event.type}`)
28return null
29}
30
31try {
32return await handler(event, client)
33} catch (err) {
34// 單一事件的錯誤不應讓整個批次失敗
35console.error(`[Router] Handler error for ${event.type}:`, err)
36return null
37}
38}
39
40module.exports = router

第 31–37 行的 try/catch 非常重要。當一個 Webhook 請求包含多個事件時， index.js 的 Promise.all(events.map(router)) 中， 若某一個事件的處理函式拋出例外，整個 Promise.all 會立即 reject， 導致其餘事件都得不到處理，且 Webhook 回傳 500 讓 LINE 誤以為請求失敗而重試。 在 router 層統一捕捉並記錄錯誤、回傳 null，可以將單一事件的失敗 隔離在自身範圍，不影響批次中的其他事件。

訊息型別的二級路由：messageHandler.js

message 事件需要再依 event.message.type 進行第二層分派。這個二級路由的邏輯與主路由相同，只是維度不同。

40image: handleImage,
41sticker: handleSticker,

49}

Lesson 2–4

Follow 事件：歡迎訊息與使用者 Profile 取得

follow 事件是你與新用戶建立關係的第一個時機。 大多數正式 Bot 都會在此時發送歡迎訊息、說明功能， 並將使用者資料儲存到資料庫，為後續的個人化互動做準備。 你可以在這個時間點呼叫 client.getProfile(userId)， 取得使用者的顯示名稱、頭像 URL 和狀態訊息。

src/handlers/followHandler.js — follow / unfollow 處理
1async function onFollow(event, client) {
2const userId = event.source.userId
3
4// 取得使用者個人資料
5 let profile
6try {
7profile = await client.getProfile(userId)
8} catch (err) {
9// getProfile 可能因隱私設定失敗，此時使用預設值
10console.warn('getProfile failed:', err.message)
11profile = { displayName: '使用者', pictureUrl: null }
12}
13
14// 將使用者資料儲存到資料庫（此處以 console.log 示意）
15console.log(`[Follow] New user: ${profile.displayName} (${userId})`)
16// await UserService.upsert(userId, { displayName: profile.displayName, ... })
17
18// 傳送歡迎訊息（最多可一次傳 5 則）
19return client.replyMessage(event.replyToken, [
20{
21type: 'text',
22text: `${profile.displayName} 你好，歡迎加入！`
23},
24{
25type: 'text',
26text: '輸入「說明」可查看所有指令，輸入「服務」可開啟選單。'
27}
28])
29}
30
31async function onUnfollow(event, client) {
32// unfollow 沒有 replyToken，不可呼叫 replyMessage
33// 此時 client 參數僅作備用，通常不使用
34const userId = event.source.userId
35console.log(`[Unfollow] User left: ${userId}`)
36// await UserService.update(userId, { isActive: false, unfollowedAt: new Date() })
37return null
38}

40module.exports = { onFollow, onUnfollow }

getProfile 的回傳物件包含以下欄位：userId、 displayName（顯示名稱）、pictureUrl（頭像 URL）、 statusMessage（狀態訊息）。pictureUrl 可能為空， statusMessage 使用者可能留白，都需要做防禦性判斷。

安全提醒 getProfile 只能在一對一聊天情境（source.type === 'user'） 可靠地呼叫。在群組情境下，若該用戶的隱私設定不允許，API 會回傳 403 錯誤。 因此呼叫 getProfile 時一定要包裹在 try/catch 中， 如範例第 6–12 行所示。
profile 物件的完整結構
userId
使用者的唯一識別碼，以 U 開頭的 33 個字元字串，在同一 Provider 下固定不變。這是儲存到資料庫時的主鍵。
displayName
使用者的 LINE 顯示名稱，使用者可隨時更改，不應作為唯一識別依據。顯示用途時請即時取得最新值。
pictureUrl
頭像圖片的 HTTPS URL，可能為 null（未設定頭像時）。直接使用此 URL 展示，不需下載。
statusMessage
使用者的狀態訊息，可能為空字串。娛樂性或個人化互動時可以讀取，但業務邏輯不應依賴此欄位。
language
使用者的 LINE 語言設定，如 zh-Hant、ja，可用於多語系 Bot 的語言偵測。
Lesson 2–5

媒體訊息下載與非同步處理策略

處理圖片、影片、音訊和檔案訊息時，面對的核心挑戰是 5 秒 Webhook 回應時限與媒體下載耗時之間的衝突。 下載一個 10MB 的影片可能需要數秒，若在回覆前等待下載完成， LINE 可能已認定 Webhook 逾時而重試請求，造成重複處理。 正確的模式是：先立即回覆一則「處理中」訊息消耗掉 replyToken， 再異步執行下載與後續處理，最後以 Push API 通知結果。

→
異步下載媒體

→
Push API 通知

媒體下載的完整實作
1const fs = require('fs')
2const path = require('path')
3
4/**
5* 下載媒體訊息並儲存到本地磁碟
6* @param {string} messageId LINE 訊息 ID
7* @param {string} ext 副檔名，如 'jpg', 'mp4', 'm4a'
8* @param {Object} client LINE SDK Client
9* @returns {string} 儲存完成的檔案路徑
10*/
11async function downloadMedia(messageId, ext, client) {
12// 確保 uploads 目錄存在
13const uploadDir = path.join(__dirname, '../../uploads')
14fs.mkdirSync(uploadDir, { recursive: true })
15
16const filePath = path.join(uploadDir, `${messageId}.${ext}`)
17
18// 從 LINE 取得媒體串流
19const stream = await client.getMessageContent(messageId)
20
21// 將串流寫入檔案（用 Promise 包裹回調式 API）
22await new Promise((resolve, reject) => {
23const writer = fs.createWriteStream(filePath)
24stream.pipe(writer)
25writer.on('finish', resolve)
26writer.on('error', reject)
27stream.on('error', reject)
28})
29
30console.log(`[Download] Saved: ${filePath}`)
31 return filePath
32}
33
34// handleImage 的完整版本（先回覆、再異步下載）
35async function handleImage(event, client) {
36const userId = event.source.userId
37const messageId = event.message.id
38
39// Step 1：立即消耗 replyToken，確保在 5 秒內回應
40await client.replyMessage(event.replyToken, {
41type: 'text', text: '已收到圖片，正在處理中，完成後將通知你。'
42})
43
44// Step 2：異步處理（不 await，讓函式立即返回）
45(async () => {
46try {
47const filePath = await downloadMedia(messageId, 'jpg', client)
48// 在此可呼叫 OCR、圖片分析等 API
49// 完成後以 Push API 推送結果
50if (userId) {
51await client.pushMessage(userId, {
52type: 'text', text: '圖片已儲存完成！'
53})
54}
55} catch (err) {
56console.error('[handleImage] Download error:', err)
57}
58)()
59}

第 45–58 行使用了一個立即執行的非同步箭頭函式（IIFE）， 這讓 handleImage 本身在 replyMessage 完成後立刻返回， 而下載工作在背景繼續執行。注意第 57 行的 catch： 背景任務的錯誤不能讓它靜默失敗，必須記錄到日誌， 否則下載失敗時你完全不會知道。

各媒體型別的副檔名對照
message.type MIME Type 副檔名 典型大小上限
image image/jpeg .jpg 10 MB
video video/mp4 .mp4 200 MB
audio audio/x-m4a .m4a 200 MB
file 依副檔名而定 參照 message.fileName 300 MB
生產環境建議 在本地開發時將媒體存到磁碟是可以的，但在雲端部署環境（如 Render、Railway）中， 容器的本地磁碟在重啟後會清空，且空間有限。生產環境應將媒體直接上傳到 物件儲存服務，例如 Google Cloud Storage、AWS S3 或 Firebase Storage， 讓 getMessageContent 的串流直接 pipe 到雲端，而不落地到磁碟。

訊息事件的子型別，如 text、image、sticker。
source.userId
用戶唯一識別碼，群組情境下可能為 undefined。
getProfile()
取得用戶顯示名稱、頭像等個人資訊。
getMessageContent()
下載媒體訊息的二進位串流。
EventRouter
依事件型別分派到對應 handler 的路由層。
dispatch table
用物件映射取代 switch，便於擴展與測試。
設計原則

單一職責
每個 handler 只負責一種事件，router 只負責分派，不混入業務邏輯。

防禦性取值
所有從 event 物件取值的操作都應考慮 undefined 情況，尤其是 source.userId。

錯誤隔離
每個事件的 try/catch 相互獨立，單一事件失敗不影響批次中的其他事件。

章節測驗
在群組中，某個成員傳訊息給 Bot。以下關於 source 欄位的描述，哪項正確？
A. source.type 為 "user"，source.userId 一定存在
B. source.type 為 "group"，source.userId 可能不存在
C. source.type 為 "group"，source.userId 一定存在
D. source.type 為 "room"，source.groupId 存在
正確。在群組（group）情境下，source.type 為 "group"， 同時提供 source.groupId。但 source.userId 依用戶的隱私設定而定， 可能不存在，取用前務必檢查。
